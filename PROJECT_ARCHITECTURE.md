## 2. How the Frontend Talks to the Backend

### Communication Pattern

The frontend uses **plain `fetch()` calls** — no WebSockets, no GraphQL, no tRPC. It's a simple REST-style JSON-over-HTTP setup.

All API calls are centralized in a single file:

📄 **`vaultrag-frontend/lib/vaultragApi.ts`**

```
Frontend (React)                        Backend (FastAPI)
─────────────────                       ─────────────────
vaultragApi.ts  ─── POST /ask ────────▶  main.py @app.post("/ask")
                ─── POST /upload-file ─▶  main.py @app.post("/upload-file")
```

### The API Base URL

```typescript
const API_BASE =
  process.env.NEXT_PUBLIC_VAULTRAG_API_URL || "http://127.0.0.1:8000";
```

The frontend reads this from an environment variable at **build time** (it's a `NEXT_PUBLIC_` variable). If unset, it defaults to `http://127.0.0.1:8000`.

### The Two Main API Functions

| Function               | HTTP Method & Endpoint | Request Body                                  | Response                                          |
|------------------------|------------------------|-----------------------------------------------|---------------------------------------------------|
| `askVaultRag()`        | `POST /ask`            | `{ query: string, user_role: "Employee" \| "Executive" }` | `{ answer: string, context_used: string[] }`      |
| `uploadExecutiveFile()`| `POST /upload-file`    | `FormData` with `file` field (PDF or CSV)     | `{ message: string }`                             |

### CORS

The backend allows all origins (`allow_origins=["*"]`), so the frontend at `localhost:3000` can talk to the backend at `localhost:8000` without issues.

### Role-Based Access Control (RBAC)

RBAC is **client-side simulated**. The user picks a role in the `Header` component (Employee or Executive). This role string is sent as `user_role` in every `/ask` request. The backend then uses it to filter which ChromaDB documents are visible:

| User Role    | Can See Documents Tagged As   | Can Upload Files? |
|--------------|-------------------------------|-------------------|
| Employee     | `Employee` only               | ❌ No              |
| Executive    | `Employee` + `Executive`      | ✅ Yes (PDF/CSV)   |

## 3. Where Documents Are Stored & Embedded

### Storage: ChromaDB (Persistent, On-Disk)

```python
chroma_client = chromadb.PersistentClient(path="./chroma_db")
collection    = chroma_client.get_or_create_collection(name="vault_docs")
```

- **Location on disk:** `./chroma_db/` directory (SQLite under the hood: `chroma_db/chroma.sqlite3`)
- **Collection name:** `vault_docs`
- **Persistence:** Data survives server restarts. ChromaDB writes to `chroma.sqlite3` automatically.

### Embedding Model: SentenceTransformer

```python
embedding_model = SentenceTransformer("all-MiniLM-L6-v2")
```

- **Model:** `all-MiniLM-L6-v2` — a lightweight, fast model that produces **384-dimensional** vectors.
- **Runs locally** in the Python process (no external API call for embeddings).
- **Loaded once at startup** and reused for every embed/query.

### How Documents Get Into ChromaDB

There are two ingestion paths:

#### Path A: Raw Text Upload (`POST /upload`)

```
Client sends { content: "some text", role: "Executive" }
    → Backend encodes text with SentenceTransformer
    → Stores in ChromaDB with metadata: { role: "Executive" }
```

#### Path B: File Upload (`POST /upload-file`)

```
Client sends PDF or CSV file via FormData
    → Backend reads file bytes (in-memory only, nothing saved to disk)
    → Extracts text:
        • PDF → uses pypdf (PdfReader) to extract page text
        • CSV → joins each row's cells with " | "
    → Splits text into ~300–500 character chunks (word-boundary aware)
    → Batch-encodes all chunks with SentenceTransformer
    → Stores each chunk in ChromaDB with metadata: { role: "Executive" }
```

### What Gets Stored Per Document Chunk

| Field       | Example                                        | Purpose                               |
|-------------|-------------------------------------------------|---------------------------------------|
| `id`        | `"a3f1b2c4-..."`                                | UUID, auto-generated                 |
| `document`  | `"Q4 revenue was 200 crore INR..."`             | The raw text chunk                   |
| `embedding` | `[0.023, -0.114, ..., 0.087]` (384 floats)     | Vector from SentenceTransformer      |
| `metadata`  | `{ "role": "Executive" }`                       | Used for RBAC filtering at query time|

## 4. How the Local Phi-3 Model Is Called

### The Chain: FastAPI → httpx → Ollama → Phi-3

The backend does **not** load Phi-3 into its own Python process. Instead, it acts as a client to **Ollama**, which is a separate service that manages model loading and inference.

```
FastAPI (main.py)
    │
    │  httpx.AsyncClient.post(OLLAMA_URL, json={...})
    │
    ▼
Ollama HTTP API  (http://localhost:11434/api/generate)
    │
    │  Loads & runs the "phi3" model locally (GPU/CPU)
    │
    ▼
Returns generated text as JSON: { "response": "The revenue is..." }
```

### The Exact HTTP Call

```python
OLLAMA_URL = os.getenv("OLLAMA_URL", "http://localhost:11434/api/generate")

async with httpx.AsyncClient(timeout=60.0) as client:
    response = await client.post(
        OLLAMA_URL,
        json={
            "model": "phi3",
            "prompt": prompt,    # ← includes context + question + instructions
            "stream": False,     # ← wait for full response, no streaming
        },
    )
```

### Key Details

| Aspect          | Detail                                                       |
|-----------------|--------------------------------------------------------------|
| **Model**       | `phi3` (Microsoft Phi-3, ~3.8B parameters)                   |
| **Streaming**   | Disabled (`stream: False`) — waits for complete response     |
| **Timeout**     | 60 seconds                                                   |
| **Error case**  | If Ollama isn't running, returns HTTP 503 to the frontend    |
| **Post-process**| `clean_llm_output()` strips prompt leakage, limits to 1-2 sentences |

### The Prompt Template

The prompt sent to Phi-3 follows a structured format:

```
---
CONTEXT:
--- Source 1 ---
[retrieved chunk 1 text]

--- Source 2 ---
[retrieved chunk 2 text]

QUESTION:
[user's original question]

INSTRUCTIONS:
* Answer using the provided context only.
* If nothing relevant is found, respond with: No relevant information found.
* Start with the direct answer in one clear sentence.
* Do not hallucinate beyond context.
---
Answer:
```

## 5. Data Flow: From "Search" Click to Answer on Screen

Here is the **complete journey** when a user types a question and presses Enter (or clicks the send button):

```
 ┌───────────────────────────────────────────────────────────────────┐
 │  STEP 1: USER INPUT                                              │
 │                                                                   │
 │  User types "What was Q4 revenue?" in InputBar                   │
 │  User role is set to "Executive" in the Header                   │
 │  User clicks Send (↑) or presses Enter                          │
 └────────────────────────────┬──────────────────────────────────────┘
                              │
                              ▼
 ┌───────────────────────────────────────────────────────────────────┐
 │  STEP 2: FRONTEND HANDLING  (app/page.tsx → handleSend)          │
 │                                                                   │
 │  a. Creates a ChatMessage { role: 'user', text: '...' }         │
 │  b. Creates (or updates) a conversation in state                 │
 │  c. Sets isTyping = true (shows SearchingAnimation)              │
 │  d. If files attached + Executive mode: uploads each via         │
 │     uploadExecutiveFile() → POST /upload-file                    │
 │  e. Calls askVaultRag(query, userRole) → POST /ask               │
 └────────────────────────────┬──────────────────────────────────────┘
                              │
     fetch("http://127.0.0.1:8000/ask", {                           
       method: "POST",                                               
       body: JSON.stringify({                                        
         query: "What was Q4 revenue?",                              
         user_role: "Executive"                                      
       })                                                            
     })                                                              
                              │
                              ▼
 ┌───────────────────────────────────────────────────────────────────┐
 │  STEP 3: BACKEND RECEIVES REQUEST  (main.py → ask_question)     │
 │                                                                   │
 │  Pydantic validates: query ≠ empty, user_role ∈ {Employee,       │
 │  Executive}                                                       │
 └────────────────────────────┬──────────────────────────────────────┘
                              │
                              ▼
 ┌───────────────────────────────────────────────────────────────────┐
 │  STEP 4: RBAC VECTOR SEARCH  (main.py → _rbac_search)           │
 │                                                                   │
 │  a. Build where-filter based on role:                            │
 │     • Employee → { "role": "Employee" }                          │
 │     • Executive → { "role": { "$in": ["Employee","Executive"] }} │
 │                                                                   │
 │  b. Encode the query with SentenceTransformer → 384-dim vector   │
 │                                                                   │
 │  c. Query ChromaDB:                                              │
 │     collection.query(                                             │
 │       query_embeddings=[vector],                                  │
 │       n_results=12,            # fetch extra for reranking       │
 │       where=where_filter,      # RBAC filter                    │
 │       include=["documents", "distances"]                          │
 │     )                                                             │
 │                                                                   │
 │  d. Filter: remove chunks with distance > 1.28 (absolute)       │
 │             or > best_distance + 0.40 (relative delta)           │
 │                                                                   │
 │  e. Rerank: prefer chunks with keyword overlap (e.g. "revenue") │
 │     Sort by (-keyword_overlap, +distance)                        │
 │                                                                   │
 │  f. Return top 4 chunks                                          │
 └────────────────────────────┬──────────────────────────────────────┘
                              │
                              ▼
 ┌───────────────────────────────────────────────────────────────────┐
 │  STEP 5: BUILD PROMPT & CALL PHI-3                               │
 │                                                                   │
 │  a. Join retrieved chunks as numbered "--- Source N ---" blocks   │
 │                                                                   │
 │  b. Build structured prompt:                                     │
 │     CONTEXT: [chunks]                                            │
 │     QUESTION: [user query]                                       │
 │     INSTRUCTIONS: [answer rules]                                 │
 │     Answer:                                                       │
 │                                                                   │
 │  c. POST to Ollama:                                              │
 │     http://localhost:11434/api/generate                           │
 │     { "model": "phi3", "prompt": "...", "stream": false }        │
 │                                                                   │
 │  d. Ollama runs Phi-3 inference → returns generated text         │
 └────────────────────────────┬──────────────────────────────────────┘
                              │
                              ▼
 ┌───────────────────────────────────────────────────────────────────┐
 │  STEP 6: CLEAN & RESPOND                                         │
 │                                                                   │
 │  a. clean_llm_output() processes the raw LLM text:              │
 │     • Extracts text after "Answer:"                              │
 │     • Drops prompt scaffolding lines (CONTEXT:, QUESTION:, etc.) │
 │     • Keeps only 1–2 sentences                                   │
 │     • Rejects if > 600 chars or still looks like leaked prompt   │
 │                                                                   │
 │  b. Returns JSON to frontend:                                    │
 │     { "answer": "Q4 revenue was 200 crore INR.",                 │
 │       "context_used": ["chunk1 text...", "chunk2 text..."] }     │
 └────────────────────────────┬──────────────────────────────────────┘
                              │
                              ▼
 ┌───────────────────────────────────────────────────────────────────┐
 │  STEP 7: FRONTEND DISPLAYS RESULT  (page.tsx → ChatWindow)      │
 │                                                                   │
 │  a. Receives { answer, context_used } from the API               │
 │  b. Creates system ChatMessage with answer text + sources        │
 │  c. Appends to conversation's messages array                     │
 │  d. Sets isTyping = false (hides SearchingAnimation)             │
 │  e. ChatWindow renders the answer bubble with "VaultRAG" label   │
 │  f. SourceSnippet components render each context chunk below     │
 │  g. If TTS is ON → SpeechSynthesis reads the answer aloud       │
 │  h. Conversation is auto-saved to localStorage                   │
 └───────────────────────────────────────────────────────────────────┘
```

### Hardcoded Constants (main.py)

| Constant                          | Value   | Purpose                                              |
|-----------------------------------|---------|------------------------------------------------------|
| `UPLOAD_MAX_BYTES`                | 25 MiB  | Max file upload size                                 |
| `UPLOAD_MAX_CHUNKS`               | 2000    | Max chunks after splitting a file                    |
| `CHUNK_MIN` / `CHUNK_MAX`        | 300/500 | Target chunk size range (characters)                 |
| `RETRIEVAL_FETCH_K`              | 12      | How many chunks to fetch from ChromaDB initially     |
| `RETRIEVAL_TOP_K`                | 4       | How many chunks to keep after reranking              |
| `RETRIEVAL_MAX_DISTANCE_ABSOLUTE`| 1.28    | Drop chunks with distance above this                 |
| `RETRIEVAL_MAX_DISTANCE_DELTA`   | 0.40    | Drop chunks much worse than the best match           |
